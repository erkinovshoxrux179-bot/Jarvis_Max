# Siri-like floating overlay widget for JARVIS

A frameless, always-on-top PyQt6 widget gives JARVIS a visual heartbeat: a pulsating orb while listening, an energetic waveform while speaking, and rotating arcs while thinking. The widget is shown on wake-word detection, transitions state in sync with the Gemini session lifecycle, and fades out after the assistant finishes speaking. The tray menu gains an "Activate JARVIS" entry that also triggers the overlay.

Watch for: **Thread-safety violation in `show_overlay`** (confirmed) ā€” the wake-word callback fires from a background thread and directly manipulates Qt widgets without marshalling to the GUI thread. **`QTimer.singleShot` from asyncio thread** (confirmed) ā€” `set_speaking(False)` is called from the asyncio event loop, and the `QTimer.singleShot` inside it is undefined behavior outside the GUI thread. **Overlay never wired to TrayService** (confirmed) ā€” the overlay is created in `main()` but never passed to the `TrayService` instance that the UI constructs internally, so the tray "Activate JARVIS" action always sees `self._overlay` as `None`.

## High-level view

The `set_state` method correctly uses a `pyqtSignal` for thread safety, but `show_overlay` and `hide_overlay` lack the same protection and are called from non-GUI threads. This is the most critical issue ā€” it will crash or silently corrupt state on some Qt builds.

The integration in `main.py` extends `JarvisLive` with an optional overlay reference. The 2-second delayed hide after speaking stops is implemented via `QTimer.singleShot` which cannot be safely called from the asyncio thread that invokes `set_speaking`. Additionally, stacked timers from rapid speak/stop transitions can dismiss the overlay mid-conversation.

The tray service gains an "Activate JARVIS" menu item and a `set_overlay` setter, but `main()` never calls `set_overlay` (and can't easily, because `TrayService` is instantiated inside `JarvisUI.__init__`). The overlay portion of the tray action is dead code.

The diff also bundles unrelated cleanup: removal of the Windows auto-startup registry logic and replacement of deprecated `audioop` with numpy RMS.

## Thread-safety gap in overlay show/hide

`show_overlay` and `hide_overlay` are called from two non-GUI contexts:

1. `_on_wake_word` in `main.py` ā€” spawned as a daemon thread by `WakeWordDetector._safe_callback`:

```python
# wake_word.py, in audio_callback:
threading.Thread(target=self._safe_callback, daemon=True).start()
```

The callback calls `overlay.show_overlay()` directly, which invokes `self.show()`, `self.move(x, y)`, and starts `QPropertyAnimation` ā€” all Qt GUI operations that must run on the main thread.

2. `set_speaking(False)` in `JarvisLive` ā€” called from the asyncio event loop running in a background thread. It calls `QTimer.singleShot(2000, self.overlay.hide_overlay)`. Creating a `QTimer` from a non-GUI thread is undefined in Qt; the timer may not fire or may crash.

The fix: add `_show_signal` and `_hide_signal` (or reuse `_state_signal` with show/hide semantics and extend `_apply_state`). The pattern already exists ā€” apply it consistently.

## Overlayā€“TrayService wiring gap

The overlay is created in `main()`:

```python
overlay = SiriOverlay()
```

`TrayService` is instantiated inside `JarvisUI.__init__` (in `ui.py` line 1060):

```python
self._tray = TrayService(self)
```

No overlay is passed. `_activate_jarvis` guards with `if self._overlay:`, so the activation path is dead code. Either `main()` needs to call `ui._tray.set_overlay(overlay)` after constructing the overlay, or the overlay needs to be created inside `JarvisUI` where `TrayService` is instantiated.

## Delayed hide can stack or misfire

When `set_speaking` transitions to `False`, it schedules `hide_overlay` after 2 seconds. If JARVIS speaks again within that window (e.g., a quick follow-up), the pending hide fires during the next speaking turn and dismisses the overlay unexpectedly. There's no cancellation token or guard.

Fix: in `hide_overlay`, check `if self._state in ("SPEAKING", "LISTENING", "THINKING"): return` to no-op when the overlay is actively in use. Or track a generation counter that the delayed callback compares against.

## Animation timer lifecycle

The `_timer` (16ms interval) starts in `__init__` and never stops. On battery-powered Windows laptops (the target platform), this prevents the process from going idle. Stop the timer in `_on_fade_out_done` and restart it in `show_overlay`.
